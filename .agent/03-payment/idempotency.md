# UniPay — Idempotency & Anti-Forgery Protocol

> **AI Instruction**: This file defines the strict idempotency requirements for the Cloudflare Worker API. Every state-changing request (POST/PUT) in the payment domain MUST require an `x-idempotency-key` header and an `x-payload-hash` header. Never bypass this layer.

---

## 1. The "Double-Charge" Problem

In a mobile-first environment (especially on campus WiFi), network drops are common. 
1. A student scans a QR code and sends 500 LKR.
2. The server processes the payment successfully.
3. The network drops *before* the server can send the "Success" `200 OK` response back to the phone.
4. The student thinks it failed and taps "Pay" again.

Without **Idempotency**, the Canteen gets 1,000 LKR, and the student is incorrectly double-charged.

---

## 2. The UniPay Solution (The Two-Key Lock)

To make an API endpoint "idempotent" means that making the same request multiple times produces the exact same result as making it once. UniPay achieves this using two mandatory parameters:

### A. The Idempotency Key (`x-idempotency-key`)
* **What it is:** A unique `UUIDv7` generated by the **Client** (the SvelteKit app or Mobile App) the moment the user opens the "Confirm Payment" screen.
* **How it works:** When the Cloudflare Worker sends this key to the database, PostgreSQL checks the `transaction_events` table. If it has seen this UUID before, it knows this is a network retry and prevents the payment from executing a second time.

### B. The Payload Hash (`x-payload-hash`)
* **What it is:** A SHA-256 hash of the exact JSON body being sent (Amount, Payee ID, etc.), generated by the Client.
* **How it works (Anti-Forgery):** If an attacker intercepts a failed request, changes the amount from `50000` to `500`, and replays it using the *same* Idempotency Key, the database will catch it. It hashes the new payload, sees it doesn't match the original hash tied to that key, and throws a `FORGERY_DETECTED` error.

---

## 3. Client-Side Implementation (SvelteKit)

The frontend team (Developer 3) must implement this logic before calling the `/api/v1/payments/transfer` endpoint.

```typescript
// SvelteKit Example: Generating the safe payload
import { v7 as uuidv7 } from 'uuid';

async function initiatePayment(payeeId: string, amountCents: number) {
    const payload = {
        payee_wallet_id: payeeId,
        amount_cents: amountCents,
        type: 'purchase'
    };
    const payloadString = JSON.stringify(payload);

    // 1. Generate the unique request ID (UUIDv7 for time-sorting)
    const idempotencyKey = uuidv7();

    // 2. Hash the exact payload string using the Web Crypto API
    const encoder = new TextEncoder();
    const data = encoder.encode(payloadString);
    const hashBuffer = await crypto.subtle.digest('SHA-256', data);
    const payloadHash = Array.from(new Uint8Array(hashBuffer))
        .map(b => b.toString(16).padStart(2, '0'))
        .join('');

    // 3. Send to Cloudflare Worker
    const response = await fetch('/api/v1/payments/transfer', {
        method: 'POST',
        headers: {
            'Content-Type': 'application/json',
            'X-Idempotency-Key': idempotencyKey,
            'X-Payload-Hash': payloadHash,
            'Authorization': `Bearer ${firebaseToken}`
        },
        body: payloadString
    });

    return response.json();
}
