# UniPay — Payment Core Architecture (Overview)

> **AI Instruction**: This directory contains the business logic, API routing, and architectural flow for moving money within the UniPay ecosystem. All logic defined here must adhere to the "Zero Trust" and "Idempotent" principles. Never trust client-side math.

---

## 1. The Payment Philosophy (Zero Trust Edge Computing)

UniPay operates on a highly distributed architecture using **Cloudflare Workers (Hono.js)** at the edge and **Neon (Serverless PostgreSQL)** as the central ledger. 

To ensure bank-grade security and handle **400,000+ transactions per day**, the Worker layer must follow these three uncompromisable rules:

1. **The Worker Does No Math:** The Worker must never calculate the final balance of a wallet. It only validates the *intent* (e.g., "Student X wants to send 500 LKR to Canteen Y") and passes it to the `process_payment()` stored procedure in the database.
2. **Strict Idempotency:** Every single payment request that hits the API must contain a UUIDv7 `idempotency_key` and a `payload_hash` generated by the client. The Worker must reject any request missing these.
3. **Fail Fast:** The Worker should check JWT validity, rate limits, and payload schema *before* opening a connection to the database to save connection pool resources.

---

## 2. High-Level Architecture Flow

When a payment is initiated, the system follows a strict, one-way execution path:

1. **The Client (SvelteKit/Mobile):** Scans a QR code, generates an Idempotency Key, hashes the payload, and sends an HTTPS POST request.
2. **The Edge API (Cloudflare Worker):** - Validates the Firebase JWT.
   - Parses the JSON payload using Zod.
   - Passes the data to Neon via connection pooling.
3. **The Vault (Neon PostgreSQL):**
   - Locks the specific wallets (`FOR UPDATE`).
   - Verifies balances.
   - Moves the money and writes the Double-Entry Ledger.
   - Returns the `transaction_event_id`.
4. **The Response:** The Worker returns a `200 OK` to the client, triggering the UI success animation.

## 4. API Security Mandates

Every endpoint in the `/api/v1/payments/*` route group MUST implement the following middleware:

* **`AuthMiddleware`**: Rejects requests without a valid Firebase JWT.
* **`RateLimitMiddleware`**: Restricts endpoints to **5 requests per second** per user to prevent basic script-kiddie attacks.
* **`ZodValidation`**: Strictly types the incoming JSON. If a client sends `amount: "500"` (String) instead of `amount: 50000` (Integer Cents), the API must throw a `400 Bad Request` instantly.
